## Thermo-nuclear code-quality audit 2026-07-01 — reconciliation + new plans

A focused audit was run against commit `c6d360f` (41 commits ahead of the
last deep audit's `8c2aec8`), combining a fresh reconciliation of plans
001–008 against current code with a strict code-quality lens (file-size
discipline, spaghetti/special-case branching, dead parallel representations,
"code judo" opportunities) applied to the areas touched by the recent shapes
drag-to-draw and checkbox→check-mark features. Three parallel read-only
subagents did the legwork; findings below were spot-checked against the
cited files before being written up.

**New plans landed from this pass**: **009** and **010** (split
`src/components/PdfCanvas.tsx`, which crossed the 1,000-line threshold —
1,061 lines as of `c6d360f` — during the recent feature work) and **011**
(retire the `form-field`/`kind: "checkbox"` path, made unreachable by the
`mark-check`/`form-mark` migration in commit `d0384b1` but never deleted).

**Reconciliation of 001–006 and 008**: none of 001–006 are done; 001–004 and
006 have drifted line-number citations (the cited code still exists and the
underlying finding is still valid, but exact line ranges moved — each plan's
own drift-check step will catch this on execution). 005 was re-confirmed
accurate as written. 006 has real, substantial progress since it was planned
(`tests/pdfEngineSave.test.ts` grew from ~4 to 8+ `describe` blocks;
`tests/exportPipeline.test.ts` now covers all four `export()` formats) but is
not complete — most `savePdf` assertions still only check
`resolves.toBeInstanceOf(Uint8Array)` rather than asserting drawn content,
and the plan's PNG-export criteria are now obsolete (`ExportFormat` dropped
`png` at some point; `exportPipeline.ts` has no PNG branch anymore — whoever
picks up 006 next should re-scope around that before starting). 008 (the
shapes-toolbar reference note) is largely reflected in shipped code already
(hint banner, `region`-placement shape tools, extensive test coverage across
`pdfCanvas.test.tsx`/`operationFactory.test.ts`/`floatingOperationToolbar.test.tsx`)
— left as RESEARCH since it's a reference note, not a plan with its own done
criteria, but no further code follow-up from it is being tracked separately.

**Spot-checked deferred items — all still present, no action taken here**:

- **BUG-01** (page-rotation overlay/coordinate desync) — confirmed still
  present; `src/utils/coordinates.ts` has no rotation math at all. The
  original deferred citation (`App.tsx:381`) is stale — `App.tsx` is now a
  23-line routing shim; the relevant code is in `PdfCanvas.tsx`'s `<Page

  account for rotation. Still the highest-value unaddressed bug; still
  deserves its own dedicated plan when picked up.
- **DEBT-05** (PdfCanvas god file) — addressed by new plans 009/010 above.
- **DX-01/DX-02** (CI doesn't run `e2e`/`format:check` despite AGENTS.md/README
  calling them gates) — confirmed still true in `.github/workflows/ci.yml`.

  `useEditorController.ts`.
- **DIR-03** (signature `drawn` mode has zero writer/overlay handling; `image`
  mode is fully wired but has no UI entry point) — confirmed still present;
  not turned into its own plan this pass (kept the plan count focused), but

**New findings not yet turned into plans** (left as future work, not
currently blocking anything):

Notable deviations/decisions made while executing:

- 006's PNG-export criteria were already obsolete per the reconciliation note
  above (`ExportFormat` dropped `png`); re-scoped around the four remaining
  formats and content-level `savePdf` assertions (decoded content-stream
  matrices/glyphs, XObject counts) instead of only `resolves.toBeInstanceOf`.
- 001's popover is a single generic `InlineInputPopover` component driven by
  a `describeInlineInput(tool, operations)` descriptor in
  `operationFactory.ts`, rather than one-off popovers per tool — every
  prompt-using tool (annotation note, link, stamp, signature, five form-field
  kinds) shares the same field-rendering/focus-trap/Escape/Enter code path.
  `createOperationsForTool` takes `resolvedFields?: Record<string, string>`
  instead of a synchronous `prompt` callback, so it stays a pure function;
  creation is deferred until the popover resolves.
- 004's `livePatch`/`liveRect` local-state pattern required a matching
  `gestureOverride()` read-path in `PdfCanvas.tsx` so the dragged/resized
  overlay still renders live during the gesture even though the committed
  `operations` array only updates once, at gesture end.

`PdfCanvas.tsx` is 588 lines after 001/004 (was 519 after 009/010/011);
`src/engine/operationWriters.ts` (002) and `src/components/InlineInputPopover.tsx`
(001) are the two other new files from this batch.
